How policy pages are generated, cached, invalidated, and served at zero marginal cost per page view.

// ISR: revalidate every 24h OR on-demand via /api/revalidate export const revalidate = 86400 // 24h in seconds export default async function PolicyPage({ params, searchParams }) { // Supabase fetch — runs at build/revalidation time only const { data: product } = await supabase .from('products').select('*') .eq('slug', params.slug).single() if (!product) notFound() // → 404 page const { data: policies } = await supabase .from('policies') .select('policy_type, content_html, version, generated_at, jurisdictions') .eq('product_id', product.id) .order('version', { ascending: false }) // Latest version of each policy type const policyMap = {} policies.forEach(p => { if (!policyMap[p.policy_type]) policyMap[p.policy_type] = p }) // Render static HTML — cached on Vercel CDN after this return <PolicyPageClient product={product} policies={policyMap} activeDoc={searchParams.doc || 'privacy'} /> }
On-demand revalidation — law changes
When a law changes, policies are regenerated and the ISR cache is purged via Next.js revalidatePath(). The next request gets a fresh page — users see the updated policy without any manual deploy.

import { revalidatePath } from 'next/cache' export async function POST(req: Request) { // Verify secret to prevent unauthorized revalidation const auth = req.headers.get('Authorization') if (auth !== \`Bearer \${process.env.REVALIDATION_SECRET}\`) { return Response.json({ error: 'Unauthorized' }, { status: 401 }) } const { slug } = await req.json() // Purge ISR cache for this specific policy page revalidatePath(\`/p/\${slug}\`) // Also revalidate any sitemap that lists policy pages revalidatePath('/sitemap.xml') return Response.json({ revalidated: true, slug, timestamp: new Date().toISOString() }) }

  • Cache hit (CDN) < 60ms

Infrastructure cost at 500 users

  • CDN bandwidth $0 (included)
  • Total infra ~$50/mo
SEO advantages of ISR
Static HTML is fully indexable by search engines. Policy pages with proper metadata (title, description, robots: noindex for policies, index for SEO landing pages) get correct crawl treatment. Sub-100ms TTFB outperforms any SSR or client-rendered alternative.
Cache invalidation strategy
24h automatic background revalidation. On-demand revalidation fires immediately after: (1) user manually regenerates a policy, (2) law_updates cron triggers regen, (3) user edits questionnaire. ISR background revalidation is non-blocking — stale content is served until fresh is ready.
